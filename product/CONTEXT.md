Where product thinking lives. Define what to build, why it matters, and what success looks like.
Specs flow downstream to `engineering/workflows/01-briefs/`.

---

## Spec Format

Every spec in `specs/` should answer:

1. **What is this?** One sentence.
2. **Why does it matter?** User problem it solves.
3. **Success metric:** How do we know it worked?
4. **Scope (MVP):** What's in. What's explicitly out.
5. **Key logic / rules:** Any business logic the engineer needs to know.
6. **Open questions:** What's still unresolved.

---

## North Star

**You sound like the best version of yourself, not like AI.**

Every feature should trace back to: does this help the user get from spoken thought to posted content faster, or does it make the output sound more authentically like them?

If a feature can't answer that question, it probably doesn't belong yet.
